Turn an Ableton Live Set into video — reading the Extensions SDK and rendering with HyperFrames

A complete, copy-along guide to an Ableton Live 12 Extension with two modes: a music-locked visualizer from the raw notes, and an AI "feedback video" where a model reviews your project and designs the clip. Both call two APIs to render. Every reader, every wall, every line that matters.

We built an Ableton Live Extension that does two things from a right-click: turn a selected clip into a beat-accurate visualizer, and generate an AI "feedback video" where a model reviews the project and designs the clip that presents its review. This post is the whole build — enough real code to recreate it, both render modes, and both API integrations. Where we hit a wall, the wall is in the text with the exact error, because you'll hit the same ones.

But Ableton is just the host we picked. HyperFrames renders any HTML that's a pure function of time, so the real subject here is a recipe — read a product's structured data, shape it, render it to an on-brand video — that drops into almost any app with a plugin or API surface. Keep that in mind while you read the SDK-specific parts; we zoom out at the end to a dozen other products where the exact same code fits.

TL;DR

An Ableton Extension is a bundled Node program that Live runs in a sandbox. You read the Set through opaque handles you rehydrate into objects; integers arrive as BigInt; there's no global menu. You shape reads into a plain Timeline JSON, feed it to a HyperFrames composition (an HTML file the renderer seeks frame-by-frame — canvas driven by an hf-seek event, or DOM driven by a paused GSAP timeline), and render. Local rendering is impossible in a real install because the Node sandbox is inherited by child processes, so the shipped path is a network render (HyperFrames Cloud / HeyGen). The AI mode adds a second API — Anthropic — for the review and for authoring the composition.

Assumed audience: you write TypeScript and have seen an esbuild config. You don't need to know HyperFrames, the Ableton Extensions SDK, or the Anthropic API — we build each up from the first call. You do need Node 20+ and, for the AI mode, an Anthropic key and a HeyGen key (they're separate accounts — more on that in Part 6).

Part 1The mental model

The one idea that explains every bug

Before any code: an Ableton Extension runs in one of two hosts, and for your code's purposes they're different machines. Almost every "works locally, breaks in Live" moment is a confusion between them.

  • The dev host — extensions-cli run, launched from your shell with Developer Mode on. Inherits your full environment. No filesystem sandbox, no Node permission model. Everything works — which is why it lies to you.
  • The managed host — what a real user gets after installing a packaged .ablx with Developer Mode off. Live spawns your Node process under a hard permission sandbox and strips a pile of web globals. This is the environment you must support.

The managed host imposes four constraints that shape the entire design. Internalize them now, and the rest of this post is obvious:

  1. Filesystem sandbox — you may read/write only two directories Live gives you (storageDirectory, tempDirectory). Not /tmp, not even your own install folder.
  2. Child processes inherit the sandbox — and can never be granted more. A spawned Node CLI is dead on arrival.
  3. Stripped web globals — fetch, Headers, Event, AbortController, TextEncoder… all missing. You polyfill before anything loads.
  4. No dynamic import() — the bundled host can't do a runtime import. Static imports only.

The build: one bundle, with a polyfill banner

esbuild produces a single CommonJS file with the SDK and all deps inlined. The crucial part is the banner — an IIFE that runs before any bundled module and restores every stripped global that Node can hand back via require. This is fix #1 for the "stripped globals" constraint.

tools/build-extension.mjsjs
import * as esbuild from 'esbuild';
import { execFileSync } from 'node:child_process';

// Inline template files so the sandboxed extension never reads its install dir
execFileSync('node', ['tools/gen-template-assets.mjs'], { stdio: 'inherit' });

// Runs before any bundled module. Restores globals from Node built-ins.
const banner = `(() => {
  const g = globalThis;
  const pick = (mod, names) => { try { const m = require(mod);
    for (const n of names) if (!g[n] && m[n]) g[n] = m[n]; } catch {} };
  pick('node:util', ['TextEncoder','TextDecoder']);
  pick('node:url', ['URL','URLSearchParams']);
  pick('node:buffer', ['Blob','File','atob','btoa']);
  pick('node:stream/web', ['ReadableStream','WritableStream','TransformStream',
    'ByteLengthQueuingStrategy','CountQueuingStrategy']);
  pick('node:perf_hooks', ['performance','PerformanceObserver']);
  pick('node:worker_threads', ['MessageChannel','MessagePort','BroadcastChannel']);
})();`;

await esbuild.build({
  entryPoints: ['src/main.ts'], outfile: 'dist/extension.js',
  bundle: true, format: 'cjs', platform: 'node',

The polyfill layer: globals with no requireable source

The banner covers globals Node can restore. A handful — Event, EventTarget, AbortController, AbortSignal, DOMException, structuredClone — are "global only" with no module to require. undici references them at load, so we shim them from npm packages. This file is imported first, before undici.

src/webglobals.tsts
import { EventTarget, Event } from 'event-target-shim';
import { AbortController, AbortSignal } from 'abort-controller';
const g = globalThis as any;

if (!g.EventTarget) g.EventTarget = EventTarget;
if (!g.Event) g.Event = Event;
if (!g.AbortController) g.AbortController = AbortController;
if (!g.AbortSignal) g.AbortSignal = AbortSignal;

if (!g.DOMException) g.DOMException = class DOMException extends Error {
  constructor(message?: string, name = 'Error') { super(message); this.name = name; }
};
if (!g.structuredClone) g.structuredClone =
  (v: unknown) => (v === undefined ? undefined : JSON.parse(JSON.stringify(v)));
src/polyfill.ts — imported on line 1 of main.tsts

Speed bump

Ordering is load-bearing

Feedback failed: Headers is not defined

If the Anthropic SDK module evaluates before these globals exist, it throws at load, not at call time. So main.ts line 1 is import './polyfill', and polyfill's line 1 is import './webglobals'. Import order in the entry file is your install order.

The entry point

activate() is called once when Live loads the extension. It binds the SDK and registers our commands. Note the first import.

src/main.tsts
import './polyfill';  // FIRST — installs globals before the SDK loads
import * as live from './liveAdapter';

export async function activate(activation: unknown): Promise<void> {
  live.bindActivation(activation);

  // Mode 1 — clip/track/arrangement → visualizer
  await live.registerStudioAction('Render Video…', 'hf.render',
    (target) => { void runStudioSession(target); });

  // Mode 2 — an arrangement selection → AI feedback video
  await live.registerStudioAction('Create Feedback Video from Selection…', 'hf.feedback',
    (target) => { void runFeedbackSession(target); }, live.PROJECT_SCOPES);
}

activate takes unknown on purpose: main.ts never imports an SDK type. That's the "iron rule" we'll enforce in Part 3 — the SDK is beta and quirky, so it lives in exactly one file.

Part 3Reading the Set

The adapter, and why it's the only SDK caller

liveAdapter.ts is the single file that imports @ableton-extensions/sdk. Everything else takes plain data. This keeps the beta surface auditable and lets you unit-test downstream code by mocking the adapter. Initialization:

liveAdapter.ts — bind oncets
import { initialize, DataModelObject, Clip, MidiClip, ClipSlot, Track, MidiTrack,
  AudioTrack, type ActivationContext, type ExtensionContext,
  type Handle, type ContextMenuScope } from '@ableton-extensions/sdk';

const API_VERSION = '1.0.0' as const; type V = typeof API_VERSION;
let ctx: ExtensionContext<V>;

export function bindActivation(activation: unknown): void {
  ctx = initialize(activation as ActivationContext, API_VERSION);
}

ctx is the root of everything: ctx.application.song, ctx.commands, ctx.ui, ctx.resources, ctx.environment, and ctx.getObjectFromHandle.

Where the user clicks: scopes

First SDK-shape surprise: there's no global menu. Every context-menu item is anchored to an object type — a scope. You register a command, then bind it to each scope where it should appear.

liveAdapter.ts — scopes & registrationts
// Clip/track scopes for a clip-specific render.
export const CLIP_SCOPES: ContextMenuScope<V>[] = [
  'MidiClip', 'AudioClip', 'ClipSlot', 'ClipSlotSelection',
  'MidiTrack', 'AudioTrack',
  'MidiTrack.ArrangementSelection', 'AudioTrack.ArrangementSelection',
];
// "Whole project" = a time selection across tracks (select-all).
export const PROJECT_SCOPES: ContextMenuScope<V>[] = [
  'MidiTrack.ArrangementSelection', 'AudioTrack.ArrangementSelection',
];

export async function registerStudioAction(title: string, commandId: string,
  onInvoke: (target: unknown) => void, scopes = CLIP_SCOPES) {
  ctx.commands.registerCommand(commandId, (...args) => onInvoke(args[0]));
  const unregs = await Promise.all(scopes.map((scope) =>
    ctx.ui.registerContextMenuAction(scope, title, commandId)));
  return () => Promise.all(unregs.map((u) => u()));  // tear-down
}
Speed bump

A Session-view clip doesn't fire the scope you'd expect

Right-clicking a clip in Session view hands you a ClipSlotSelection, not a MidiClip. We registered on MidiClip, tested in Arrangement view, and the item was silently missing in Session view. Register the ClipSlot scopes too and unwrap the clip from the slot (below).

Turning a click into a selection

The callback's argument type depends on the scope: a Handle for object scopes, or a selection object for the selection scopes. A handle is opaque — getObjectFromHandle(handle, Class) rehydrates it into a live object. Here's the full resolver, branching on runtime shape:

liveAdapter.ts — getSelection()ts
export interface SelectionContext {
  scope: 'clip' | 'track' | 'arrangement';
  clipName: string; clipColor: string; isMidi: boolean;
  startBeat: number; durationBeats: number;
  clip?: Clip<V>; tracks?: Track<V>[];
}

export async function getSelection(targetArg: unknown): Promise<SelectionContext> {
  // (a) Arrangement time selection: has selected_lanes + a time window.
  if (isArrangementSelection(targetArg)) {
    const start = num(targetArg.time_selection_start);
    const end   = num(targetArg.time_selection_end);
    const tracks = targetArg.selected_lanes.map((h) => ctx.getObjectFromHandle(h, Track<V>));
    return { scope: 'arrangement', clipName: tracks[0]?.name ?? 'Arrangement',
      clipColor: '#ff5722', isMidi: tracks.some((t) => t instanceof MidiTrack),
      startBeat: start, durationBeats: end - start, tracks };
  }
  // (b) Session clip-slot selection: unwrap the first slot holding a clip.
  if (isClipSlotSelection(targetArg)) {
    for (const h of targetArg.selected_clip_slots) {
      const slot = ctx.getObjectFromHandle(h, ClipSlot<V>);
      if (slot.clip) return clipSelection(slot.clip);
    }
    throw new Error('No clip in the selected clip slots.');
  }
  // (c) Object scope: it's a Handle — rehydrate, then discriminate by class.
  const obj = ctx.getObjectFromHandle(targetArg as Handle, DataModelObject<V>);
  if (obj instanceof Clip) return clipSelection(obj);
  if (obj instanceof Track) {
    const clips = obj.arrangementClips;
    const start = clips.length ? Math.min(...clips.map((c) => num(c.startTime))) : 0;
    const end   = clips.length ? Math.max(...clips.map((c) => num(c.endTime)))   : 0;
    return { scope: 'track', clipName: obj.name, clipColor: '#ff5722',
      isMidi: obj instanceof MidiTrack, startBeat: start,
      durationBeats: end - start, tracks: [obj] };
  }
  throw new Error(`Unsupported target: ${obj?.constructor?.name}`);
}

const isArrangementSelection = (x: any) => !!x && 'selected_lanes' in x;
const isClipSlotSelection   = (x: any) => !!x && 'selected_clip_slots' in x;

The BigInt trap

Reading is pleasant — clip.notes, note.pitch, song.tempo — until you serialize. Then:

Speed bump

Integers cross the bindings as BigInt, even where the types say number

TypeError: Cannot mix BigInt and other types, use explicit conversions

First seen on Clip.color, then note times/pitch/velocity. The TypeScript types claim number; the runtime hands you a bigint. It survives until arithmetic or JSON.stringify, then throws far from the read that caused it. The fix is a discipline: coerce every integral read at the boundary.

liveAdapter.ts — the coercions everything routes throughts
function num(v: number | bigint | undefined, fallback = 0): number {
  return v === undefined ? fallback : Number(v);
}
function colorToHex(c: number | bigint): string {
  return '#' + (num(c) & 0xffffff).toString(16).padStart(6, '0');
}
function clampVelocity(v: number | bigint | undefined): number {
  return Math.min(127, Math.max(1, Math.round(num(v, 100))));
}

Reading notes

The core reader. Note times are clip-relative beats; we offset them so the exported region starts at beat 0. Muted notes are skipped; probability is passed through when present.

liveAdapter.ts — getNotes()ts
export async function getNotes(sel: SelectionContext): Promise<Note[]> {
  if (!sel.isMidi) return [];
  const out: Note[] = [];
  if (sel.clip instanceof MidiClip) {
    collectClipNotes(sel.clip, 0, undefined, out);           // single clip
  } else for (const track of sel.tracks ?? []) {          // arrangement window
    if (!(track instanceof MidiTrack)) continue;
    for (const clip of track.arrangementClips) {
      if (num(clip.endTime) <= sel.startBeat) continue;
      if (num(clip.startTime) >= sel.startBeat + sel.durationBeats) continue;
      if (clip instanceof MidiClip)
        collectClipNotes(clip, num(clip.startTime) - sel.startBeat, trackId(track), out);
    }
  }
  return out.sort((a, b) => a.startBeat - b.startBeat);
}

function collectClipNotes(clip: MidiClip<V>, offset: number, tid: string|undefined, out: Note[]) {
  for (const n of clip.notes) {
    if (n.muted) continue;
    out.push({ pitch: num(n.pitch), startBeat: num(n.startTime) + offset,
      lengthBeats: num(n.duration), velocity: clampVelocity(n.velocity),
      ...(n.probability !== undefined ? { probability: num(n.probability) } : {}),
      ...(tid ? { trackId: tid } : {}) });
  }
}

What the SDK doesn't give you (yet)

The section we wish we'd read first. Beta 1.0.0 exposes a solid core, but several things you'll reach for aren't there. Design around them from the start.

No automationNo API to read envelopes/automation curves. getAutomation() returns {}. If your visual rides a filter sweep, you can't see it — derive motion from notes.
Static tempo onlysong.tempo is one value; no tempo-map readout. We build a one-point map, so beat↔second is exact only at constant tempo.
No song time signatureExposed on scenes, not the song. We default to 4/4.
No track colorOnly clips carry color. We read the first in-range clip's color as the track's identity.
Object-anchored menusNo global command surface — every action is a right-click on a specific object type.
The upside

What is there is plenty: notes (pitch/start/length/velocity/probability), per-clip color, per-track device names, cue points as section markers, renderPreFxAudio to bounce audio, and importIntoProject to drop the finished MP4 back into the Set. That last pair is what makes it a loop, not an export.

Part 4The data contract

One JSON, then no more SDK

We never hand raw SDK objects to the renderer. Every read is assembled into a plain Timeline — serializable, testable, SDK-free. The exporter is pure orchestration (mock the adapter and it unit-tests cleanly):

types.ts — the contract (excerpt)ts

exporter.tsts
export async function exportSelection(sel: live.SelectionContext, req: RenderRequest) {
  const [tempoMap, timeSignatures, notes, automation, markers, tracks] = await Promise.all([
    live.getTempoMap(), live.getTimeSignatures(), live.getNotes(sel),
    live.getAutomation(sel), live.getMarkers(sel), live.getTracks(),
  ]);
  const workDir = req.outputDir!;
  await fs.mkdir(workDir, { recursive: true });

  // Try to bounce audio; null → render silent (visuals are pure fn of the timeline)
  const audioPath = await live.bounceAudio(sel, path.join(workDir, 'audio.wav'));
  const durationSeconds = new TimeBridge(tempoMap).beatsToSeconds(sel.durationBeats);

  const timeline: Timeline = {
    formatVersion: '1.0.0',
    meta: { title: sel.clipName, clipColor: sel.clipColor,
      exportedAt: new Date().toISOString(), sourceScope: sel.scope },
    timing: { durationBeats: sel.durationBeats, tempoMap, timeSignatures },
    audio: { file: 'audio.wav', durationSeconds, offsetBeats: 0 },
    notes, markers, tracks,
    video: { ...ASPECTS[req.aspect], fps: req.fps, style: req.style },
  };
  await fs.writeFile(path.join(workDir, 'timeline.json'), JSON.stringify(timeline, null, 2));
  return { timeline, audioPath, workDir };
}

Beats ↔ seconds: the one time conversion

HyperFrames seeks by seconds; music is in beats. TimeBridge is the single implementation of that math (two implementations = drift), anchored so beat 0 === second 0 at the region start. At constant tempo, it's a scale; the ramp path integrates properly. It compiles for Node and the browser, so the composition uses the same code to place notes.

timebridge.ts — the core conversionsts
export class TimeBridge {
  constructor(private map: TempoPoint[]) {
    this.map = [...map].sort((a, b) => a.beat - b.beat);
    // precompute cumulative seconds at each tempo point for O(log n) lookups
  }
  beatsToSeconds(beat: number): number {
    const i = this.segmentAt(beat);
    return this.cumSec[i] + segmentSeconds(this.map[i], this.map[i + 1], beat);
  }
  secondsToBeats(t: number): number { /* inverse; inverts the ramp integral when ramping */ }
}
// Seconds spanned from a segment's start to `beat`. Constant tempo → linear;
// ramping tempo → the log integral of a linearly-varying BPM.
function segmentSeconds(a: TempoPoint, b: TempoPoint | undefined, beat: number): number {
  const span = Math.max(0, beat - a.beat);
  if (a.ramp && b && b.bpm !== a.bpm) {
    const k = (b.bpm - a.bpm) / (b.beat - a.beat);      // bpm per beat
    return (60 / k) * Math.log((a.bpm + k * span) / a.bpm);
  }
  return (60 / a.bpm) * span;
}
Subtle bug we fixed

Our binary search for the active tempo segment was originally capped at length-2. That silently extended the second-to-last segment's tempo (or worse, its ramp) past the end of the map. segmentAt must be allowed to return the last point — past it, segmentSeconds holds a constant BPM because map[i+1] is undefined.

Part 5Mode 1 · Track → Video

A HyperFrames composition is an HTML file it seeks

The renderer loads your HTML, then seeks it frame by frame: it sets a virtual time and captures the pixels. The rule that makes music-sync exact: every pixel is a pure function of time. No requestAnimationFrame state, no accumulators, no Date.now(). Same t in, same frame out — so the render is note-accurate by construction, and safe to capture out of order across parallel workers.

Two ways to receive that clock. Canvas compositions get an hf-seek event (detail.time in seconds) — that's Mode 1. DOM compositions expose a paused GSAP timeline the renderer advances — that's Mode 2. Here's the canvas wiring:

The data arrives two ways: the render inlines it as window.TIMELINE (no runtime fetch — the sandbox forbids it), while the dev preview falls back to fetching timeline.json. renderFrame reads the timeline and draws — here's the shape, pure in t:

pulse-waveform — renderFrame(t), abridgedjs
let T = null, bridge = null, peaks = null;
async function init() {
  T = window.TIMELINE ?? await (await fetch('./timeline.json')).json();
  bridge = new TimeBridgeLib.TimeBridge(T.timing.tempoMap);
  canvas.width = T.video.width; canvas.height = T.video.height;
  peaks = await computePeaks('./' + T.audio.file, 768).catch(() => new Float32Array(768));
}

// Energy of notes striking around this beat — drives the pulse. Pure in t.
function noteEnergy(beat) {
  let e = 0;
  for (const n of T.notes) {
    const dt = beat - n.startBeat;
    if (dt < 0 || dt > 1.5) continue;
    e += (n.velocity / 127) * Math.exp(-dt * 6);      // exponential decay, deterministic
  }
  return Math.min(e, 2);
}

function renderFrame(t) {
  if (!T || !peaks) return;
  const beat = bridge.secondsToBeats(t);       // seconds → musical position
  const energy = noteEnergy(beat);
  // clear, draw a beat-synced ring, a scrolling waveform, note-pulse dots…
  for (const n of T.notes) {                      // one dot per active note
    const dt = beat - n.startBeat; if (dt < 0 || dt > 1) continue;
    const x = W * (0.1 + 0.8 * ((n.pitch % 36) / 36));   // x by pitch
    const r = (n.velocity / 127) * 14 * (1 - dt);           // size by velocity decay
    ctx.globalAlpha = 1 - dt; ctx.beginPath(); ctx.arc(x, y, r, 0, Math.PI * 2); ctx.fill();
  }
}

Staging the bundle

Before rendering, the composition's files are written into the work dir. Because the sandbox forbids reading our own install dir, template files are inlined into the bundle at build time (a codegen step produces TEMPLATE_ASSETS) and written from memory. We also patch the root's data-* from the timeline and inline the data as JS.

render.ts — stageBundle()ts
export async function stageBundle(job: RenderJob): Promise<void> {
  if (job.prestaged) return;   // mode 2 writes a complete comp itself
  const style = path.basename(job.templateDir);
  const assets = TEMPLATE_ASSETS[style];        // inlined at build time, from memory
  for (const [name, content] of Object.entries(assets))
    await fs.writeFile(path.join(job.workDir, name), content);

  // HyperFrames reads size/duration from the HTML, not CLI flags — patch them.
  const idx = path.join(job.workDir, 'index.html');
  const { width, height } = job.timeline.video;
  const html = (await fs.readFile(idx, 'utf8'))
    .replace(/data-duration="[^"]*"/, `data-duration="${job.timeline.audio.durationSeconds.toFixed(3)}"`)
    .replace(/data-width="[^"]*"/,  `data-width="${width}"`)
    .replace(/data-height="[^"]*"/, `data-height="${height}"`);
  await fs.writeFile(idx, html);
  await fs.writeFile(path.join(job.workDir, 'timeline.js'),
    `window.TIMELINE = ${JSON.stringify(job.timeline)};`);
}

Part 6Mode 2 · Feedback Video (API #1)

A compact summary, not the raw notes

Mode 2 sends the model a small, structured description of the selection — counts, per-track devices/density, register, sections. No raw notes, no audio. Cheap, and enough to reason about arrangement and balance.

liveAdapter.ts — getProjectSummary(), abridgedts
export async function getProjectSummary(sel: SelectionContext): Promise<ProjectSummary> {
  const song = ctx.application.song;
  const scopeTracks = sel.tracks?.length ? sel.tracks : [...song.tracks];
  const start = sel.startBeat, end = start + sel.durationBeats;
  const tracks: TrackSummary[] = [];
  let totalNotes = 0, minPitch = Infinity, maxPitch = -Infinity;

  for (const track of scopeTracks) {
    let noteCount = 0, color;
    for (const clip of track.arrangementClips) {
      if (num(clip.endTime) <= start || num(clip.startTime) >= end) continue;
      color ??= colorToHex(clip.color);
      if (clip instanceof MidiClip) for (const n of clip.notes) {
        if (n.muted) continue;
        noteCount++; const p = num(n.pitch);
        minPitch = Math.min(minPitch, p); maxPitch = Math.max(maxPitch, p);
      }
    }
    totalNotes += noteCount;
    tracks.push({ name: track.name, kind: track instanceof MidiTrack ? 'midi' : 'audio',
      color, noteCount, devices: track.devices.map((d) => d.name),
      density: +(noteCount / (end - start)).toFixed(2) });
  }
  return { title: scopeTracks.length === song.tracks.length ? 'Full Project' : 'Arrangement Selection',
    tempoBpm: num(song.tempo, 120), timeSignature: '4/4', durationBeats: end - start,
    scope: sel.scope, tracks, totalNotes,
    pitchRange: Number.isFinite(minPitch) ? { min: minPitch, max: maxPitch } : null };
}

import Anthropic from '@anthropic-ai/sdk';
const MODEL = 'claude-opus-4-8';

export async function generateFeedback(summary: ProjectSummary, apiKey: string): Promise<FeedbackReport> {
  const client = new Anthropic({ apiKey });
  const stream = client.messages.stream({
    model: MODEL,
    max_tokens: 8000,
    thinking: { type: 'adaptive' },                          // let the model decide depth
    system: SYSTEM,                                          // "you are a mixing engineer…"
    output_config: { format: { type: 'json_schema', schema: FEEDBACK_SCHEMA } },
    messages: [{ role: 'user',
      content: 'Review this Ableton project and return JSON.\n\n' + JSON.stringify(summary, null, 2) }],
  });
  const message = await stream.finalMessage();
  if (message.stop_reason === 'refusal') throw new Error('The model declined to review this project.');
  const text = message.content.find((b) => b.type === 'text')?.text;
  const report = JSON.parse(text!) as FeedbackReport;
  report.score = Math.max(0, Math.min(100, Math.round(report.score)));   // clamp for the gauge
  return report;
}
API note

On Opus 4.6+ models, thinking is {type:'adaptive'} — the old budget_tokens form is rejected. Always check stop_reason === 'refusal' before reading content: a safety decline is an HTTP 200 with an empty content array, not an exception.

Where the conventions come from: install the HyperFrames skills

The rules in the system prompt below aren't invented — they're distilled from the HyperFrames skills. HyperFrames ships a CLI (npx hyperframes) that does more than render: it installs a set of authoring skills — Markdown playbooks that teach an AI the composition contract, the GSAP runtime, and the determinism rules. That same CLI is where hyperframes lint and hyperframes render come from.

the HyperFrames toolchain — one CLIbash
npx hyperframes skills     # install authoring skills for AI coding tools
npx hyperframes init       # scaffold a composition project
npx hyperframes add data-chart   # pull a prebuilt block from the registry
npx hyperframes lint DIR --json  # validate a composition (our fix loop uses this)
npx hyperframes render DIR       # local render (dev host only)

npx hyperframes skills drops skills like hyperframes-core (the composition contract), hyperframes-animation (the GSAP/canvas runtime), and hyperframes-registry into your agent's skill directories (~/.claude/skills, ~/.agents/skills). Reading hyperframes-core is exactly how we learned the non-negotiables — the sized root, the single paused timeline, the determinism bans — and the frozen-render bug below is precisely what the skill warns about.

Design choice

We distilled the skill into the prompt, not dumped it

Rather than pasting a multi-thousand-word skill into every request, we lifted only the hard rules that break a render if violated and kept the system prompt tight — cheaper and more deterministic output. If you'd rather, you can hand the model the skill file directly and let it read the full contract; the trade-off is prompt size vs. control. Either way, the skills are the source of truth — write them down once, from the CLI, before you prompt.

Claude authors the composition — then lints and fixes it

Here's the fun part. Instead of a fixed template, the model writes the HyperFrames composition from the review, following those conventions. The system prompt is the composition contract (the hard rules the renderer needs), and GSAP is provided locally so there's no CDN. Then we lint the output — with the same hyperframes lint from the CLI above — and feed errors back for repair, up to three rounds, before rendering.

export async function authorFeedbackComposition(report, summary, apiKey, workDir, w, h) {
  const client = new Anthropic({ apiKey });
  await fs.writeFile(path.join(workDir, 'gsap.min.js'), GSAP_MIN);   // bundled, no CDN

  let html = await authorHtml(client, report, summary, w, h);       // first draft
  await fs.writeFile(path.join(workDir, 'index.html'), html);

  for (let round = 1; round <= 3; round++) {
    const errors = (await lint(workDir)).filter((f) => f.severity === 'error');
    if (!errors.length) return { html, durationSeconds: readDuration(html) };
    html = await fixHtml(client, html, errors);                     // feed errors back
    await fs.writeFile(path.join(workDir, 'index.html'), html);
  }
  throw new Error('Composition still failing lint');   // caller falls back to fixed template
}

async function lint(workDir: string) {                          // `hyperframes lint --json`
  const out = await runCapture('npx', ['-y','hyperframes','lint',workDir,'--json'], workDir);
  try { return JSON.parse(out.slice(out.indexOf('{'))).findings ?? []; } catch { return []; }
}

Part 7Rendering (API #2)

Why the shipped renderer has to be a network call

We had local rendering working on the dev host — spawn the hyperframes CLI, it drives headless Chrome and ffmpeg, out comes an MP4. Then we installed the real .ablx and hit the wall.

Speed bump · the architectural one

The Node sandbox is inherited by child processes — and can't be widened

npx exited with 1 … ERR_ACCESS_DENIED, permission: 'FileSystemRead', resource: '.../npm/bin/npx-cli.js'

The managed host runs your Node under --permission (fs limited to storage/temp). Any child Node process inherits that and can never get more access. hyperframes is a Node CLI, so it can't even read its own program files. Stripping NODE_OPTIONS doesn't help — the inheritance is intrinsic. Native binaries (ffmpeg, zip) are fine as children; Node CLIs are not. So production rendering must be a network render: zip the composition, upload, poll, download.

We keep renderLocal for the dev host, and default to cloud everywhere else:

render.ts — renderLocal (dev only)ts
export async function renderLocal(job: RenderJob): Promise<string> {
  const out = path.join(job.workDir, 'output.mp4');
  await stageBundle(job);
  await run('npx', ['-y','hyperframes','render', job.workDir,
    '--output', out, '--fps', String(job.timeline.video.fps), '--quiet'], job.workDir);
  return out;
}

API #2 — HyperFrames Cloud: upload, submit, poll, download

HyperFrames Cloud is a HeyGen-hosted service (api.heygen.com), authed with an x-api-key. The whole render is: zip the work dir, upload the zip, submit a render referencing it, poll, download the MP4. The upload is the part we got wrong first.

Speed bump

Don't invent an API shape the official CLI never uses

HTTP 400 invalid_parameter — "Field required", param: "project.base64.media_type"

We tried submitting the project as inline base64 and kept 400-ing on a field we were sending. The tell: the official CLI has zero references to that field — it never uses inline base64. It uploads via a direct-to-S3 flow and references the result by asset_id. We replicated that exactly and it went through first try.

render.ts — the direct-to-S3 upload the CLI usests
async function uploadProjectZip(zipBytes: Buffer, apiKey: string, signal?): Promise<string> {
  const checksum = createHash('sha256').update(zipBytes).digest('hex');

  // 1. Ask for a presigned upload slot.
  const init = await api('POST', '/v3/assets/direct-uploads', apiKey, {
    filename: 'project.zip', content_type: 'application/zip',
    size_bytes: zipBytes.byteLength, checksum_sha256: checksum }, signal);
  const d = init.data ?? init;

  // 2. PUT the bytes straight to S3 — NO api key on this request.
  const headers = { 'content-type': 'application/zip', ...d.upload_headers };
  const put = await fetch(d.upload_url, { method: 'PUT', headers,
    body: new Uint8Array(zipBytes), signal });
  if (!put.ok) throw new Error(`upload PUT failed: HTTP ${put.status}`);

  // 3. Finalize (retry 409 = not yet visible after the PUT).
  for (let i = 0; i < 5; i++) {
    try { await api('POST', `/v3/assets/${d.asset_id}/complete`, apiKey, { checksum_sha256: checksum }, signal);
          return d.asset_id; }
    catch (e) { if (i === 4 || !/HTTP 409/.test(String(e))) throw e; await sleep(500 * (i + 1)); }
  }
}
render.ts — renderCloud(): submit, poll, downloadts
export async function renderCloud(job, apiKey, onProgress?, signal?): Promise<string> {
  await stageBundle(job);
  await run('zip', ['-r','-q', zipPath, '.'], job.workDir);   // native binary — allowed
  const assetId = await uploadProjectZip(await fs.readFile(zipPath), apiKey, signal);

  const submit = await api('POST', '/v3/hyperframes/renders', apiKey, {
    project: { type: 'asset_id', asset_id: assetId },       // the proven shape
    fps: job.timeline.video.fps, format: 'mp4', title: job.timeline.meta.title }, signal);
  const renderId = submit.render_id ?? submit.data?.render_id;

  for (;;) {                                                 // poll until done
    await sleep(5000, signal);
    const st = await api('GET', `/v3/hyperframes/renders/${renderId}`, apiKey, undefined, signal);
    const s = st.status ?? st.data?.status;
    if (s === 'completed') {
      const res = await fetch(st.video_url ?? st.data.video_url, { signal });
      await fs.writeFile(outFile, Buffer.from(await res.arrayBuffer()));
      return outFile;
    }
    if (s === 'failed') throw new Error('Cloud render failed');
    onProgress?.('rendering', Number(st.progress ?? 0));
  }
}

The api() helper is a thin fetch wrapper that sets x-api-key and surfaces the response body on error — which is how we saw that 400 field name instead of a generic failure. And run() spawns children with a widened PATH, strips NODE_OPTIONS, and captures output so a child failure tells you why:

render.ts — api() + run()ts
async function api(method, apiPath, apiKey, body?, signal?) {
  const res = await fetch(API_BASE + apiPath, { method, signal,
    headers: { 'x-api-key': apiKey, ...(body ? { 'content-type': 'application/json' } : {}) },
    body: body ? JSON.stringify(body) : undefined });
  const text = await res.text();
  if (!res.ok) throw new Error(`HyperFrames ${method} ${apiPath} → HTTP ${res.status}: ${text}`);
  return JSON.parse(text);
}

function run(cmd, args, cwd): Promise<void> {
  const env = { ...process.env, PATH: extraToolDirs + ':' + process.env.PATH };
  delete env.NODE_OPTIONS;   // don't hand the sandbox flags to children
  return new Promise((resolve, reject) => {
    let out = ''; const child = spawn(cmd, args, { cwd, env });
    child.stdout.on('data', (d) => out += d); child.stderr.on('data', (d) => out += d);
    child.on('exit', (code) => code === 0 ? resolve()
      : reject(new Error(`${cmd} exited ${code}. Output tail:\n${out.slice(-1800)}`)));
  });
}
Not a bug — a billing fact

The two APIs have separate wallets

HTTP 402 insufficient_credit — "At least 9.0 API credits are required to render"

The review (Anthropic) and the render (HeyGen) are separate services with separate billing. Money in your Anthropic account does nothing for HeyGen rendering, and vice versa. Surface this in your UI, or users will file it as a bug. Which means you need to store two keys.

Two keys, resolved and stored

Each API's key resolves from: an explicit value → an env var → a file in storageDirectory (the only writable place). If none, we prompt and persist. Same shape for both.

export async function resolveApiKey(src: { key?: string; storageDir?: string }) {
  if (src.key) return src.key.trim();

}

Where to start

Don't start from a blank folder — there's a minimal starter template in the repo that already does the whole loop (read a clip's notes → a canvas composition → render → import), in about ten commented files. Clone it, run npm run check, and build up from there.

If you're building from scratch anyway, the order that would have saved us the most time: (1) scaffold the polyfill layer and an offline load-check on day one; (2) build liveAdapter as the single SDK file and route every integer through num(); (3) decide your scopes and prototype the menu placement before any data work; (4) target cloud rendering from the start, treat local render as a dev convenience; (5) if you add the AI mode, expect two separate API bills and store two keys.

The SDK is beta, and it shows — but the primitives are right, and importIntoProject closing the loop is what makes an extension feel native rather than like an export button. Everything above is the tax to get there. Now you don't pay it twice.

Part 10Beyond Ableton

The same recipe, other products

Strip the Ableton specifics and what's left is a four-step recipe: read a host's structured data → shape it → author or pick an HTML composition → render it deterministically. Only the first step is Ableton-shaped. HyperFrames itself only asks for HTML that's a pure function of time, so the input can be any structured data and the host can be any product with a plugin or API surface.

That gives you two independent axes of flexibility. The host is whatever holds the data — a DAW, an IDE, a storefront, a dashboard, a wearable. And the composition can be a fixed template, a registry block, or — as in Mode 2 — authored per item by a model, so every video is bespoke and on-brand without a designer in the loop.

Why the HTML/CSS/JS foundation is what makes this easy

None of that flexibility is free — it falls out of one architectural choice: a composition is a web page, not a proprietary project format. That single fact is why integration is cheap:

  • No new language, and it's the one everyone already knows. Not a node graph or a compositing DSL — HTML/CSS/JS. The talent pool is the whole web. It's also exactly why Mode 2 works: authoring a composition is authoring a webpage, the most-represented thing in a model's training data, so an LLM does it reliably and a linter can check it. A proprietary format would make "AI designs the video" a research project.
  • The web ecosystem is your component library. GSAP, D3, Three.js, Lottie, Chart.js, SVG, WebGL, web fonts — anything that renders in a browser renders in a composition. You don't rebuild charts or easing; you import them. Your existing brand CSS, tokens, and logos drop in directly, so "on-brand" is free.
  • The hard parts of motion graphics come for free. Text wrapping, flexbox/grid layout, i18n, emoji, responsive units — the browser already solves them. A native engine reimplements every one. Change the aspect ratio and CSS just reflows.
  • The integration seam is "emit a string," not "link an SDK." A host only has to produce HTML. Every language can do that — Node, Rails, Python, a serverless function. Data binding is a JSON global (window.TIMELINE = {…} — literally what this extension does). No native library, no ABI, no graphics context.
  • The preview loop is a browser refresh, and compositions are just text files — they version-control, diff, code-review, and read transparently, which is what makes them lintable and closes the Mode-2 author→lint→fix loop.
  • The renderer already exists everywhere. HyperFrames seeks headless Chromium — ubiquitous, free, battle-tested — so it never ships a bespoke renderer into each host. "Seek to time t, screenshot" maps perfectly onto a browser you've written as a pure function of time.

The through-line: HTML/CSS/JS turns video from a specialized native-graphics problem into a web problem — so the existing web workforce, web libraries, browsers, and LLMs are all instantly pointed at it. A few concrete places the exact same code drops in:

That's the whole point of building on HyperFrames: the renderer is the boring, solved part, so the interesting work is only ever "what data, and what should it look like."
